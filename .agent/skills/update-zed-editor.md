# Skill: Updating the Zed Editor Ebuild

This document describes the canonical procedure for updating the `app-editors/zed` ebuild to the latest upstream release. Follow every step in order, every time.

> **Safety rules**: Refer to `.agent/instructions/general.md` for general overlay safety rules and constraints before making any changes.

**Package directory**: `app-editors/zed/`
**Package metadata**: `.agent/packages.json` (entry for `app-editors/zed`)
**Upstream repo**: `zed-industries/zed` on GitHub

## Scope of Changes

- Only create/modify `.ebuild` files and update `Manifest` within `app-editors/zed/`
- Do not leave temporary scripts, scratch files, or extra documentation in the overlay
- Use a temporary working directory (e.g. `/tmp/zed-update-work/`) for any intermediate files and clean it up when done
- **Always check for dependency changes before considering the update complete** â€” never skip this step

## Step-by-Step Update Process

### 1. Determine Versions

Query the latest upstream release and compare it to the current overlay version.

```bash
# Get latest upstream version
LATEST=$(curl -s https://api.github.com/repos/zed-industries/zed/releases/latest \
  | grep '"tag_name"' | sed -E 's/.*"v([^"]+)".*/\1/')

# Get current overlay version
CURRENT=$(ls -1 app-editors/zed/zed-*.ebuild \
  | grep -oP '\d+\.\d+\.\d+' | sort -V | tail -1)

echo "Current: ${CURRENT}"
echo "Latest:  ${LATEST}"
```

Alternatively, run `scripts/check-updates.sh` from the overlay root â€” it reads `.agent/packages.json` and reports whether `app-editors/zed` has an update available.

If the versions are equal, the overlay is already up to date. **Stop here.**

### 2. Verify Sources Are Available

Both the main source tarball and the crates tarball must exist before proceeding.

```bash
# Main source tarball
curl -sfI "https://github.com/zed-industries/zed/archive/refs/tags/v${LATEST}.tar.gz" \
  && echo "âś“ Main source" || echo "âś— Main source NOT available â€” abort"

# Crates tarball (generated by gentoo-crate-dist, may lag by hours)
curl -sfI "https://github.com/gentoo-crate-dist/zed/releases/download/v${LATEST}/zed-${LATEST}-crates.tar.xz" \
  && echo "âś“ Crates tarball" || echo "âś— Crates tarball not yet available â€” wait and retry"
```

- If the **main source** is missing, the tag does not exist upstream yet. Wait and retry.
- If only the **crates tarball** is missing, it typically appears within a few hours. Wait and retry.

### 3. Copy the Ebuild to the New Version

```bash
cd app-editors/zed

FROM_VER=$(ls -1 zed-*.ebuild | grep -oP '\d+\.\d+\.\d+' | sort -V | tail -1)
TO_VER="${LATEST}"

echo "Bumping: v${FROM_VER} -> v${TO_VER}"
cp "zed-${FROM_VER}.ebuild" "zed-${TO_VER}.ebuild"
```

### 4. Check for Dependency Changes (MANDATORY)

Always diff the upstream `Cargo.toml` between the old and new versions. This is the most critical step â€” most Zed releases change at least a few git dependency revisions.

```bash
cd app-editors/zed

FROM_VER=$(ls -1 zed-*.ebuild | grep -oP '\d+\.\d+\.\d+' | sort -V | tail -2 | head -1)
TO_VER=$(ls -1 zed-*.ebuild | grep -oP '\d+\.\d+\.\d+' | sort -V | tail -1)

echo "Diffing Cargo.toml: v${FROM_VER} -> v${TO_VER}"

diff \
  <(curl -s "https://raw.githubusercontent.com/zed-industries/zed/v${FROM_VER}/Cargo.toml") \
  <(curl -s "https://raw.githubusercontent.com/zed-industries/zed/v${TO_VER}/Cargo.toml")
```

#### What to Look for in the Diff

##### A) GIT_CRATES â€” Git-Sourced Dependencies

Each `GIT_CRATES` entry has this format:

```
[crate-name]='https://github.com/owner/repo;COMMIT_HASH;repo-dirname-%commit%/optional/subpath'
```

| Diff pattern | Required action |
|---|---|
| A git dep's `rev =` changed | Update the commit hash in `GIT_CRATES` |
| A new `{ git = "...", rev = "..." }` dep appeared | Add a new entry to `GIT_CRATES` |
| A git dep was removed | Remove its entry from `GIT_CRATES` |
| A crates.io dep version changed | No action â€” handled by the crates tarball |

**New workspace members**: If the diff shows new entries in `members = [...]`, fetch the `Cargo.toml` for each new member crate and scan it for git dependencies. These may introduce new `GIT_CRATES` entries.

When a repository is renamed or its commit hash changes, update **ALL** `GIT_CRATES` entries that reference that repository (many repos provide multiple sub-crates from a single repo, e.g. `livekit-rust-sdks`, `python-environment-tools`, `xim-rs`).

##### B) WEBRTC_COMMIT

If the livekit-rust-sdks commit hash changed in `GIT_CRATES`, verify whether `WEBRTC_COMMIT` needs updating. Check the releases page at:

`https://github.com/livekit/rust-sdks/releases`

The `WEBRTC_COMMIT` value must match a release tag of the form `webrtc-XXXX` at that URL.

##### C) RUST_MIN_VER

Look for changes to `rust-version` in the `Cargo.toml` diff. If it changed, update `RUST_MIN_VER` in the ebuild.

##### D) System-Level DEPEND/BDEPEND

Scan the release notes for any new system library requirements. If new native dependencies are needed, add them to `DEPEND` and/or `BDEPEND`.

### 5. Apply Any Required Ebuild Changes

Edit the new ebuild (`zed-${TO_VER}.ebuild`) directly. Common edits include:

- Update `GIT_CRATES` commit hashes (keep entries in alphabetical order)
- Add or remove `GIT_CRATES` entries
- Update `WEBRTC_COMMIT`
- Update `RUST_MIN_VER`
- Update `src_prepare()` sed substitution commits/variables if the corresponding `GIT_CRATES` entries changed (e.g. `CALLOOP_COMMIT`, `NOTIFY_COMMIT`, `LIVEKIT_COMMIT`, etc.)
- Add/remove system dependencies in `DEPEND`/`BDEPEND`

**Important**: The ebuild uses `${PV}` for version references â€” never hardcode the version number inside the ebuild.

### 6. Generate the Manifest

```bash
cd app-editors/zed
pkgdev manifest
```

This downloads all sources referenced in `SRC_URI` and regenerates the `Manifest` file with their checksums. If this fails, the most likely causes are:

- Wrong commit hash in `GIT_CRATES` (download 404)
- Force-pushed upstream repo (hash mismatch)
- `WEBRTC_COMMIT` doesn't match an actual release tag

### 7. Lint the Ebuild (Optional but Recommended)

Run the overlay's lint script from the repo root:

```bash
scripts/lint.sh app-editors/zed
```

This runs `pkgcheck` inside the testenv container and reports QA issues.

### 8. Clean Up and Report

Remove the temporary working directory if one was created.

Report what was done:
- Version bumped from X to Y
- Any `GIT_CRATES` changes (list each: added, removed, or updated with oldâ†’new hash)
- Any `WEBRTC_COMMIT` changes
- Any `RUST_MIN_VER` changes
- Any `DEPEND`/`BDEPEND` changes
- Any other ebuild modifications

## GIT_CRATES Format Reference

```bash
declare -A GIT_CRATES=(
    [crate-name]='https://github.com/owner/repo;FULL40CHARHASH;repo-name-%commit%'
    [sub-crate]='https://github.com/owner/repo;FULL40CHARHASH;repo-name-%commit%/path/to/sub-crate'
    [crate-a]='https://github.com/owner/monorepo;FULL40CHARHASH;monorepo-%commit%/crates/crate-a'
    [crate-b]='https://github.com/owner/monorepo;FULL40CHARHASH;monorepo-%commit%/crates/crate-b'
)
```

The cargo eclasses use this map to:

1. Download `https://github.com/owner/repo/archive/HASH.tar.gz`
2. Unpack it as `repo-name-HASH/` (with `%commit%` substituted for the actual hash)
3. Make the crate available to the offline build

When a repo is renamed or its hash changes, update **ALL** entries that reference it. For example, if the `livekit-rust-sdks` commit changes, every crate sourced from that repo (`livekit`, `libwebrtc`, `livekit-api`, `livekit-protocol`, `livekit-runtime`, `webrtc-sys`, `webrtc-sys-build`) must be updated to the new hash.

## Common Issues

| Issue | Cause / Fix |
|---|---|
| Crates tarball 404 | Normal for brand-new releases. The `gentoo-crate-dist` bot generates it asynchronously. Wait a few hours and retry. |
| WebRTC prebuilt not found | `WEBRTC_COMMIT` does not match any release tag at `livekit/rust-sdks`. Check the releases page and update. |
| `pkgdev manifest` fails | Usually a wrong commit hash in `GIT_CRATES`, or a force-pushed repo where the hash no longer exists. |
| `pkgdev manifest` re-downloads everything | Expected when hashes changed â€” it needs to fetch the new archives. |

## Key Facts

- Zed releases frequently (often multiple times per week)
- Most releases change at least a few git dependency revisions â€” always diff `Cargo.toml`
- The ebuild uses `${PV}` â€” never hardcode the version inside the ebuild
- Crates.io dependency version bumps require no ebuild changes (the crates tarball handles them)
- The `src_prepare()` function contains sed substitutions that rewrite git deps to local paths; if a git dep's commit hash changes, the corresponding variable in `src_prepare()` must also be updated
